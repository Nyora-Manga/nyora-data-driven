# Engine verification — live end-to-end

Generated by the `verifyAll` harness (`src/main/kotlin/app/nyora/data/runtime/VerifyAll.kt`,
`./gradlew verifyAll --args="<maxCandidates> <perCallTimeoutMs> [engineКeys,csv]"`). For every
bundled engine it builds a `SourceDef` generically from `repo/<engine>.json` and runs the real
`getPopular(0) → getDetails(first) → getPageList(firstChapter)` pipeline against the live site,
trying candidates until one completes. Run date: 2026-07-13, from a slow + Cloudflare-IP-blocked
local network (this matters — see "blocked" below).

### 🐞 Bug found and fixed
- **pizzareader** — `getString()` on numeric JSON fields threw `JSONException` on instances that
  return bare numbers: `adult` (int `0`) and `rating` (BigDecimal `9.14`). Both reads now go
  through `optInt` / `opt(...).toString().toFloatOrNull()`, tolerant of int **or** string.
  → now PASS (FmTeam). This was the ONLY genuine engine defect surfaced across all 34 engines.

### 🚧 Blocked by environment — NOT proven buggy (10)
Zero of these threw a code-level exception; they returned empty (a Cloudflare challenge page
parses to nothing) or were unreachable. Verifying them needs the CF-solving context the
production helper provides (`solveAntiBot` is a stub in this prototype) and/or a non-CF-blocked IP.
- **Cloudflare-walled** reachable sources (confirmed via direct probe — `"Just a moment…"`):
  animebootstrap (all 3 live sources CF), gattsu, wpcomics, asurascans (its 1 live source
  AsuraComic is CF + a bespoke Next.js API), and most mangareader/onemanga/madara *candidates*
  (the engines still PASSed once the harness reached a non-CF source).
- **Dead / tiny / DNS-fail** reachable sources: scan (scanita/mangafr/manga-terra), mangabox
  (no reachable pure-config source), cupfox (`UnknownHostException`), batoto (`SocketTimeout`).
- **sinmh** — list parses (ykmh → 36 popular) but its single live source returns 0 chapters
  (selector variance or a CF-walled detail page); only 1 live source, can't generalize.
- **heancmsalt** — all 3 rows are `broken:true`, so nothing to test.

## Notes
- The harness classifies UNREACHABLE (network) vs EMPTY_* (reached, empty) vs PARSE_FAIL (engine
  threw a non-network exception = a real bug). Only PARSE_FAIL indicates an engine defect.
- EMPTY_POPULAR for an engine with a large catalogue (e.g. madara/mangareader) is a candidate-luck
  artifact: the first N `broken:false` rows happened to be CF-walled/dead. Raising `maxCandidates`
  reached a working source and flipped them to PASS.
- Every one of the 702 `broken:false` rows maps to a bundled engine (0 unrenderable); the 6
  pagedmanga/initmanga base-class rows with no engine were marked `broken:true`.
- Full verification of the CF-walled engines belongs to the production integration (the helper
  solves Cloudflare) or a re-run from an unblocked IP — not a prototype gap.
